Reconciliation test flow — Cases 6–8

sequenceDiagram
    participant T as pytest
    participant HS as httpserver :9090
    participant RECON as reconciliation Lambda
    participant DDB1 as access-requests

    RECON->>JSTUB: lambda.invoke(violation_payload)
    JSTUB->>DDB2: put_item
    RECON-->>T: report
    deactivate RECON

    T->>DDB2: scan()
    T->>T: assert violation type + user

What the httpserver intercepts

X-Amz-Target headerReturns
CloudTrail_20131101.StartQuery {"QueryId": "test-q-001"}
CloudTrail_20131101.GetQueryResults {"QueryStatus":"FINISHED","QueryResultRows":[...], "QueryStatistics":{}}
Two StartQuery + GetQueryResults pairs per reconciliation invocation: one for the S3 native event data store, one for the curated (Vault/DB) store. Register 4 ordered handlers per test case.

QueryResultRows format

Each row is a list of single-key dicts — one per column: 
[
  [
    {"eventTime":  "2024-01-15 10:30:00"},
    {"eventName":  "GetObject"},
    {"eventSource":"s3.amazonaws.com"},
    {"principalId":"ABCDE:alice@example.com"},
    {"reqBucketName":"example-data-bucket"},
    {"reqObjectKey": "customer/data.csv"}
  ]
]
The reconciliation Lambda flattens each row into a plain dict keyed by column name. Use principalId with a colon-separated email suffix to simulate human S3 access.
